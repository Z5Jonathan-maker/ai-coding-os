The Eden CRM "Scales" tool compares a carrier's Xactimate estimate vs the PA's to build a supplement. Repo `Z5Jonathan-maker/ClaimPilot` (`/tmp/unified` = working clone), backend `services/{pdf_parser,estimate_matcher,scope_gap_engine}.py`, route `routes/scales.py`, UI `frontend/src/components/Scales.jsx`. See [[project-eve-self-updating-brain]] for deploy mechanics (CI-gated, autoDeploy OFF). Built/pivoted 2026-06-03.

**KEY ARCHITECTURE DECISION (the pivot): stop chasing line-by-line 1:1 matching as the product.** Research (3 parallel research-scout briefs + GitHub recon, 2026-06-03) established WHY perfect matching is impossible from PDFs and what to do instead:
- A Xactimate line's true identity is its code: **CAT + SEL + ACT** (e.g. `RFG IWS &` = ice-&-water-shield, remove-&-replace). **The PDF strips these codes entirely** — so any PDF-to-PDF matcher (incl. LEVLR, the #1 commercial tool at $399/mo) is forced to match on description TEXT = inherently lossy. Codes live in the **ESX file** (just a ZIP of XML; rename .esx→.zip, unzip). ESX-to-ESX would match near-exactly, but carriers send PDFs ("good luck getting an ESX from an adjuster" — LEVLR's own FAQ), so ESX ingestion is a "when available" bonus, NOT the foundation.
- **No OSS does two-estimate comparison.** Best OSS parser = `rpriven/xactparse` (single-file regex PDF→xlsx, multi-pattern w/ AGE/LIFE/CONDITION depreciation + `[M]` marker — worth mining if parser needs hardening). `Elliott1985/claim-integrity-engine` = single-estimate rule audit. Our two-estimate comparator is genuinely novel.
- **The expert PA workflow is NOT line-first:** price-list/O&P/depreciation audit → category subtotal scan → line deltas → quantity check → price check. The tool now mirrors this.

**What's solid (don't redo):**
- **Parsing is bulletproof.** `pdf_parser.py` uses coordinate-based columnar reconstruction (pymupdf `get_text("words")` grouped by y), header-aware format detection for TWO layouts: PA/Care-Claims `DESC QTY UNIT REMOVE REPLACE TAX O&P TOTAL` and carrier/People's-Trust `DESC QTY UNIT UNIT-PRICE TAX RCV DEPREC ACV`. Reconciliation guardrail: parsed sum must equal the PDF's stated "Replacement Cost Value" within max($1,1%); flags instead of showing garbage. Both real Lewis estimates reconcile to the penny ($29,703.53 carrier / $41,023.53 PA). Category is re-inferred from the COMPLETE description AFTER continuation lines append (a row's first line often lacks the keyword, e.g. "Remove Laminated - comp." before "shingle rfg." wraps → was mis-filed GEN).
- **`scope_gap_engine.py` is the headline deliverable** (deterministic, citation-backed, matching-free): O&P omission (~20%, cites Trinidad v. Florida Peninsula 121 So.3d 433 (Fla. 2013)), missing categories ($0 carrier vs PA), category shortage (carrier under-scoped a category it included), roof-square shortage, commonly-missed accessories (starter, ridge cap, drip edge, I&WS, step flashing, valley metal, pipe jack, ridge vent, permit, dump, detach-reset gutters/satellite) each with FBC/statute citation (R905.2.8.5 drip edge mandatory; R905.1 mfr-required starter/IWS; 626.9744 matching). On Lewis pair: 6 findings, ~$18.8K defensible. Returned in `POST /compare` as `scope_gaps`; shown as lead "Supplement Findings" card.

**Line matcher (`estimate_matcher.py`)** is now a reviewable ASSIST, not the headline: global best-first assignment, similarity = exact-code→1.0 else desc 0.55 + cat 0.25 + unit 0.20, abbreviation/synonym map + token singularization. ~24/32 on Lewis with all high-dollar pairs correct; remaining unmatched are mostly GENUINE scope differences, not bugs. Do NOT over-tune it against one file (overfitting trap that ate most of 2026-06-03).

**FL legal anchors for supplements** (verified 2026-06-03, extends [[reference-fl-pa-law-verified-2026-06]]): O&P = Trinidad "reasonably likely a GC needed" (3-trade rule is carrier negotiating position, not law); Fla. Stat. 626.9744 matching (⚠ ~25 FL carriers now have an endorsement capping matching at 1% Cov A — check the policy); 627.7011 L&O default 25% dwelling; SB 2A (2023): supplement window 18 months from DOL, carrier pays/denies in 60 days, AOB banned post-2023. Labor depreciation generally improper but site-specific → flag for human review, never auto-assert.

**⛔ NO ESX EVER** — user ruled out ESX ingestion permanently (2026-06-03). Do not propose or build it again. PDF-only.

**Supplement letter (BUILT 2026-06-03):** `POST /api/scales/dispute-letter` → `services.ai_analyzer.generate_supplement_letter` drafts a claim-aware demand letter on top of `scope_gaps` findings. It is **claim-grounded the same way Eve chat is**: pulls full claim file via `routes/ai/shared.fetch_claim_context(claim_id, user)` + `format_claim_context_for_prompt` (claim record + notes + docs + comms) — claim_id comes from request or the comparison's stored `claim_id`. Adds FL statute context via `get_florida_statute_context`. Runs on `claude_max.complete` (Opus, Max subscription); deterministic draft fallback if offline. Never invents claim facts (bracketed placeholders when no claim linked). UI "Generate Supplement Letter" button passes `activeComparison.claim_id`.

**Rebuttal email (BUILT 2026-06-03):** `POST /dispute-letter` takes `mode` ('letter' = formal supplement demand; 'rebuttal' = reply to carrier email thread). Rebuttal pulls the FULL carrier thread off the linked claim (`db.communications`, untruncated bodies, 15 msgs oldest→newest — NOT the 150-char-truncated `recent_communications` in format_claim_context_for_prompt) and `services.ai_analyzer.generate_rebuttal_email` drafts a reply rebutting the carrier's specific points, each tied to a scope-gap finding + citation + $. UI: "Rebuttal Email" button (requires linked claim) + "Generate Supplement Letter" button. Claim selector: auto-detects comparison's claim_id, overridable via dropdown of all claims (`GET /api/claims/`); `letterClaimId` state drives both buttons.
